Введение:

Данный документ описывает технический дизайн обучающей платформы "Киберпес Цербер". Он детализирует архитектуру системы, выбранные технологии, структуру данных, дизайн RAG-системы, технические аспекты пользовательского интерфейса и анализа данных, а также рассматривает вопросы развертывания, безопасности, масштабируемости и сопровождения.

Этот документ предназначен для разработчиков и всех заинтересованных сторон, чтобы обеспечить общее понимание технической реализации проекта и служить руководством для разработки.

1. Обзор архитектуры системы

"Киберпес Цербер" будет реализован как автономное десктопное приложение (на первом этапе прототипа). Основные компоненты системы и их взаимодействие представлены на следующей схеме (диаграмму можно вставить сюда, если есть, или добавить описание):

Основные компоненты:

  • Пользовательский интерфейс (UI): Отвечает за взаимодействие с пользователем, отображение задач, визуализацию Цербера, получение ввода пользователя и отображение результатов. Реализован на Pygame.

  • Логика приложения (Application Logic): Ядро приложения, управляющее игровым процессом, обучением, задачами, прогрессом пользователя, системой наград и взаимодействием с RAG-системой. Реализовано на Python.

  • Система обучающих задач (Task System): Хранит определения и логику обучающих задач различных типов (Распознавание атак, Выбор защиты, Анализ логов). Данные задач могут храниться в [JSON] файлах.

  • RAG-система (Retrieval-Augmented Generation System): Интеллектуальный помощник, предоставляющий ответы на вопросы пользователя, используя базу знаний по AI-безопасности. Состоит из компонентов:

    • Индексатор (Indexer): Обрабатывает и индексирует базу знаний для эффективного поиска.

    • Поисковик (Retriever): Ищет релевантные документы в индексе на основе запроса пользователя.

    • Генератор (Generator): Формирует ответ на основе найденных документов и запроса пользователя.

  • База данных обучения (Learning Data - в виде файлов на первом этапе): Хранит данные о прогрессе обучения пользователей, решенных задачах, наградах и т.д. На первом этапе прототипа данные могут храниться в CSV файлах.

  • Модуль анализа данных и отчетности (Data Analysis & Reporting): Отвечает за анализ данных обучения и формирование отчетов, в том числе в формате Excel. Реализован на Python с использованием библиотек pandas, openpyxl.

  • Ресурсы (Assets): Хранит статические ресурсы приложения: изображения, звуки, шрифты и т.д.

Взаимодействие компонентов:

  1. Пользователь взаимодействует с UI.

  2. UI передает действия пользователя в Логику приложения.

  3. Логика приложения управляет игровым процессом, взаимодействует с Системой обучающих задач для предоставления задач пользователю.

  4. При необходимости, Логика приложения обращается к RAG-системе для получения ответов на вопросы пользователя.

  5. RAG-система использует Индексатор, Поисковик и Генератор для обработки запроса и формирования ответа на основе Базы знаний.

  6. Логика приложения обновляет Базу данных обучения (файлы) с информацией о прогрессе пользователя.

  7. Модуль анализа данных и отчетности обрабатывает данные из Базы данных обучения и формирует отчеты, которые могут быть экспортированы в Excel.

  8. UI отображает информацию пользователю, включая задачи, визуализацию Цербера, прогресс, ответы RAG-системы и отчеты.

  9. Ресурсы используются всеми компонентами для визуализации, звукового сопровождения и т.д.

2. Технологический стек

  • Язык программирования:

    • Python (версия 3.8+): Выбран в качестве основного языка программирования благодаря своей универсальности, богатой экосистеме библиотек для UI, Machine Learning, Data Science и простоте разработки. Python обеспечивает быструю разработку прототипа и позволяет эффективно реализовать все необходимые компоненты системы.

  • Пользовательский интерфейс (UI):

    • Pygame:

      • Pygame: Рассматривается для создания более визуально привлекательного и динамичного 2D UI с анимациями Цербера. Подходит для геймифицированных приложений.

  • RAG-система:

    • Библиотеки Python для NLP и Machine Learning:

      • Transformers (Hugging Face): Для загрузки и использования предварительно обученных языковых моделей (BERT, RoBERTa).

      • Sentence-Transformers: Для создания векторных представлений (embeddings) текста, необходимых для семантического поиска в RAG.

      • FAISS (Facebook AI Similarity Search): Для эффективного индексирования и быстрого поиска ближайших соседей в векторном пространстве (для поиска релевантных документов в базе знаний).

      • LangChain: Фреймворки для упрощения построения RAG-пайплайнов (опционально, можно реализовать RAG и "вручную" для прототипа).

  • База данных обучения (на этапе прототипа - файловое хранилище):

    • Формат файлов: CSV файлы для хранения данных о прогрессе пользователей, задачах и т.д. Простота реализации и достаточная функциональность для прототипа.

    • Библиотеки Python для работы с файлами: csv встроенные библиотеки Python.

  • Анализ данных и отчетность:

    • Pandas: Библиотека Python для анализа и обработки данных, эффективная работа с табличными данными.

    • Openpyxl: Библиотека Python для работы с файлами Excel (.xlsx), для создания отчетов в формате Excel.

    • Matplotlib: Библиотеки Python для визуализации данных (создание графиков и диаграмм - опционально на первом этапе, если потребуется визуализация данных внутри приложения).

  • Система контроля версий:

    • Git: Для контроля версий исходного кода, совместной разработки и отслеживания изменений.

    • GitHub (или GitLab, Bitbucket): Для хостинга репозитория кода и совместной работы.

  • Документация:

    • Confluence: Для создания и ведения проектной документации (как и данный документ).

    • Markdown: Для хранения текстовой документации в репозитории кода (README.md, файлы в docs/).

3. Структура данных и хранение

3.1. Данные обучающих задач:

  • Задачи будут храниться в файлах формата [JSON] в директории data/tasks/.

  • Каждый файл будет содержать массив задач определенного типа (например, attack_recognition_tasks.json).

  • Структура данных для каждой задачи (пример для "Распознавание атак"):

[
  {
    "task_id": "attack_recog_1",
    "task_type": "attack_recognition",
    "level": "beginner",
    "question_text": "Следующий текстовый запрос к AI-модели: '...",
    "data": "...", // Данные задачи (текст, изображение, код и т.д.)
    "correct_answer": "yes", // Или "no", или название типа атаки
    "explanation": "Объяснение правильного ответа и принципов атаки",
    "hints": ["Подсказка 1", "Подсказка 2"]
  },
  // ... другие задачи ...
]

  • Аналогичная структура будет использоваться для задач "Выбор защиты" и "Анализ логов", с соответствующими полями question_text, data, correct_answer, explanation, hints и специфическими полями, если необходимо.

3.2. Данные прогресса пользователя:

  • Данные о прогрессе пользователей будут храниться в [CSV или JSON] файле в директории data/. Например, user_progress.csv.

  • Структура данных для записи прогресса пользователя:

user_id,task_id,task_type,answer,is_correct,time_spent,score_earned,timestamp
user_1,attack_recog_1,attack_recognition,yes,true,15,10,2025-03-20 10:00:00
user_1,defense_select_2,defense_selection,method_A,false,25,0,2025-03-20 10:00:25
// ... другие записи ...

  • Этот файл позволит отслеживать прогресс обучения каждого пользователя, анализировать успешность решения задач и формировать отчеты.

3.3. База знаний RAG:

  • База знаний для RAG-системы будет состоять из текстовых файлов, содержащих:

    • Документацию проекта (экспорт из Confluence в текстовом формате).

    • Описания задач (из файлов data/tasks/).

    • Теоретические материалы по AI-безопасности (текстовые файлы, статьи, выдержки из учебников).

  • Файлы базы знаний будут храниться в директории data/rag_knowledge_base/.

  • Формат файлов: Markdown (.md) или простые текстовые файлы (.txt). Будет проведена предварительная обработка текста для оптимизации индексации RAG.

4. Дизайн RAG-системы

4.1. Компоненты RAG-системы:

  • Индексатор (Indexer):

    • Использует библиотеку sentence-transformers для создания векторных представлений (embeddings) для каждого текстового фрагмента в базе знаний.

    • Использует библиотеку FAISS для создания индекса векторной базы данных, обеспечивающего быстрый поиск ближайших соседей.

    • Индексация будет выполняться офлайн (заранее, перед запуском приложения) при первом запуске или при обновлении базы знаний.

    • Индекс будет сохранен на диск для последующей загрузки и использования.

  • Поисковик (Retriever):

    • Получает запрос пользователя в текстовом виде.

    • Использует ту же модель sentence-transformers, что и индексатор, для создания векторного представления запроса.

    • Использует индекс FAISS для поиска k-наиболее релевантных фрагментов текста из базы знаний, векторные представления которых наиболее близки к вектору запроса (семантический поиск ближайших соседей).

    • Возвращает найденные фрагменты текста (или ссылки на них) в качестве контекста.

  • Генератор (Generator):

    • Получает запрос пользователя и контекст, найденный Поисковиком.

    • Использует предварительно обученную языковую модель из библиотеки transformers (distilbert-base-uncased-distilled-squad - question answering) для генерации ответа на вопрос пользователя, опираясь на предоставленный контекст.

    • Возвращает сгенерированный текстовый ответ.

    • На первом этапе прототипа может использоваться модель для Question Answering, которая извлекает ответ из контекста. В дальнейшем можно рассмотреть генеративные модели для более свободной генерации ответов.

4.2. Процесс работы RAG-системы (поток данных):

  1. Индексация (офлайн):

    • База знаний (текстовые файлы) -> Индексатор -> Векторные представления (embeddings) -> Индекс FAISS -> Сохраненный индекс.

  2. Запрос пользователя (во время работы приложения):

    • Запрос пользователя (текст) -> Поисковик -> Вектор запроса -> Поиск в индексе -> Релевантные фрагменты текста (контекст).

    • Запрос пользователя (текст) + Контекст -> Генератор -> Сгенерированный ответ (текст).

    • Ответ -> UI для отображения пользователю.

4.3. Обучение RAG-системы ("настройка" и индексация):

  • "Обучение" RAG-системы в данном проекте заключается в:

    • Выборе и настройке предварительно обученных моделей (sentence-transformers для embeddings, языковая модель для генерации/question answering).

    • Подготовке и обработке базы знаний (текстовые файлы, форматирование, разделение на фрагменты).

    • Индексации базы знаний с помощью Индексатора и сохранения индекса.

    • Тестировании и оценке качества ответов RAG-системы и, при необходимости, корректировке параметров и настроек.

  • В данном прототипе не планируется дообучение языковых моделей на специфических данных. Фокус на интеграции и демонстрации концепции RAG с использованием готовых моделей.

    • Взаимодействие с логикой приложения: UI будет взаимодействовать с модулями логики приложения (logic/) через четко определенные интерфейсы (например, вызовы функций, обработка событий).

6. Анализ данных и отчетность - технические аспекты

  • Сбор данных:

    • Модули логики приложения будут записывать данные о процессе обучения (как описано в разделе "Структура данных и хранение") в CSV файл (user_progress.csv или аналогичный).

  • Анализ данных:

    • Модуль data_analysis/ будет содержать скрипты Python для анализа данных.

    • Библиотека pandas будет использоваться для загрузки, обработки и анализа данных из файла user_progress.csv.

    • Будут реализованы функции для расчета основных статистических показателей (средний процент правильных ответов, распределение времени, прогресс обучения и т.д.).

    • Библиотеки matplotlib или seaborn могут быть использованы для создания визуализаций данных (опционально на первом этапе).

  • Формирование отчетов в Excel:

    • Модуль data_analysis/excel_reporting.py будет использовать библиотеку openpyxl для создания отчетов в формате Excel (.xlsx).

    • Отчеты будут включать таблицы со статистическими данными и, возможно, простые графики (если будет реализована визуализация данных).

    • Будет реализована функция экспорта отчетов в файлы Excel, которые пользователь сможет открыть и просмотреть.

  • Для более масштабного развертывания в будущем (если проект будет развиваться) можно рассмотреть варианты веб-приложения (с использованием легковесного веб-фреймворка Flask).

8. Безопасность (технические аспекты AI-безопасности)

  • Обучающий контент: Технический дизайн обеспечивает возможность гибкого добавления и обновления обучающих задач и теоретических материалов, что позволяет поддерживать актуальность контента по AI-безопасности.

  • RAG-система как инструмент обучения безопасности: RAG-система сама по себе является инструментом для повышения осведомленности пользователей о вопросах AI-безопасности, предоставляя доступ к знаниям и ответам на вопросы.

  • Имитация атак и защит: Обучающие задачи будут технически реализованы таким образом, чтобы имитировать различные типы атак на AI/ML системы и демонстрировать принципы работы методов защиты. Фокус на концептуальном представлении и обучении пониманию принципов, а не на реальном моделировании сложных кибератак.

  • Безопасность кода платформы (основы): При разработке кода будут соблюдаться базовые принципы безопасного программирования, хотя для прототипа фокус больше на функциональности и демонстрации концепции, чем на продвинутой безопасности самой платформы.

9. Масштабируемость и сопровождение

  • Модульная архитектура: Разделение системы на модули (UI, Logic, Tasks, RAG, Data Analysis) способствует масштабируемости и упрощает сопровождение. Отдельные модули можно разрабатывать, тестировать и обновлять независимо.

  • Использование библиотек и фреймворков: Выбор популярных и хорошо поддерживаемых библиотек Python (Pygame, transformers, pandas, openpyxl и т.д.) облегчает сопровождение и расширение функциональности проекта.

  • Структурированный код и документация: Соблюдение принципов чистого кода, комментирование кода и ведение подробной документации (в Confluence и в коде) упростят сопровождение проекта в будущем.

  • Возможность расширения базы знаний RAG: Дизайн RAG-системы предусматривает возможность добавления новых текстовых файлов в базу знаний, что позволяет легко расширять охват тем и обновлять информацию.

  • Для реальной масштабируемости и развертывания веб-приложения или мобильного приложения потребуется пересмотр архитектуры и технологий, но принципы модульности и структурированности, заложенные на этапе прототипа, будут полезны и в дальнейшем развитии проекта.

10. Заключение

Данный документ "Технический дизайн" описывает архитектуру, технологии и основные технические решения, заложенные в основу обучающей платформы "Киберпес Цербер". Технический дизайн направлен на создание функционального прототипа, демонстрирующего интерактивное обучение основам AI-безопасности с использованием RAG-системы и геймификации. Выбранные технологии и модульная архитектура обеспечивают гибкость разработки, возможность расширения функциональности и демонстрируют навыки в использовании современных инструментов и подходов к разработке программного обеспечения и AI-систем. В процессе разработки технический дизайн может быть уточнен и адаптирован при необходимости.
